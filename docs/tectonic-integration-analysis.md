# Tectonic for a Rust LaTeX→Typst Converter: Full Feature Report and Integration Analysis

## TL;DR
- **Tectonic is a self-contained, embeddable Rust/C TeX engine (forked from XeTeX, using TeX Live, MIT-licensed) that compiles LaTeX all the way to PDF/XDV. It is the wrong tool for "unfold all packages and macros into a normalized LaTeX token stream":** it exposes no expanded-but-still-LaTeX intermediate representation — its only intermediate outputs (XDV/SPX) are geometry-level glyph/box streams that have already discarded `\section`, environments, and math semantics.
- **For the preprocessing step you actually want, build on a real LaTeX parser. The single best-fit reference is the Rust `mitex-parser` crate (lossless `rowan` syntax tree, the same tech rust-analyzer uses) and the `tylax` converter built on it, which already implements a token-based macro expander for `\newcommand`/`\def`/`\let`/`\ifmmode`.** Complement with `latexpand`/`de-macro` for file/`\input` flattening, and reserve your SKILL.md AI layer for genuine corner cases.
- **Tectonic still earns a place in your pipeline as a validation oracle, not a parser:** use its embeddable `tectonic::latex_to_pdf` library API to confirm the input compiles before conversion, to fetch package sources on demand via its bundle system, and (optionally) to render both original LaTeX and generated Typst to PDF for visual round-trip comparison.

## Key Findings
1. Tectonic is a modernized, complete, self-contained TeX/LaTeX engine powered by XeTeX and TeX Live, delivered both as a CLI (`tectonic`) and as a set of reusable Rust crates. Current released crate is version 0.16.9; license is MIT.
2. Tectonic's distinctive features: zero-config operation, automatic on-demand download of support files/packages from a versioned "bundle" with local caching, reproducible builds, a XeTeX-based engine producing PDF (via xdvipdfmx) or XDV/SPX, a "V2" cargo-like CLI anchored on `Tectonic.toml`, a `watch` command, BibTeX (now pure-Rust) and external biber integration, opt-in shell-escape, and an experimental SPX→HTML pathway.
3. Tectonic produces **no expanded-LaTeX intermediate**. Its pipeline runs the full TeX interpreter; the earliest accessible output is XDV (or SPX), which is a DVI-derivative describing positioned glyphs, boxes, fonts, and rules — semantic structure such as sections, environments, and math meaning is gone by then.
4. The deeper, fundamental obstacle is intrinsic to TeX, not Tectonic: TeX macro expansion is Turing-complete and intertwined with execution and mutable category codes (catcodes), so "expand all macros to recover clean LaTeX" is not a well-defined operation in general.
5. The strongest reference and reusable building block for your tool is the Rust `mitex`/`mitex-parser` family (and the `tylax` converter built on it), which parse LaTeX to a lossless `rowan` syntax tree and already do real macro expansion. Other relevant approaches: Pandoc's LaTeX reader, the LaTeXML Perl reimplementation of TeX, `unified-latex` (JS), and `latexpand`/`de-macro` for flattening.

## Details

### Part 1 — Full Tectonic feature inventory

**What it is and its lineage.** Tectonic is "a modernized, complete, self-contained TeX/LaTeX engine, powered by XeTeX and TeXLive." Its own crate documentation describes it as "a complete TeX/LaTeX engine converted into a standalone library… derived from the XeTeX variant of TeX and uses the support files packages by the TeX Live project." More precisely, it is "forked from the XeTeX extension to the classic 'Web2C' implementation of TeX." The core engine is a fork of XeTeX's C sources (e.g. `tex.rs` calls into `tex_simple_main` in `core-bridge.c`, which calls `tt_run_engine` in `xetex-ini.c`), and the project tracks upstream XeTeX via a `reference_sources` submodule. PDF generation is handled by an embedded `xdvipdfmx` driver, exactly as in a normal XeTeX→PDF flow.

**Maintainership, status, license.** The project lead is Peter Williams (pkgw); the crate is authored "by Peter Williams, Rune Tynan, Mark and 51 contributors" per the crates.io/lib.rs listing, with crate ownership also including CraftSpider and rm-dr. The current released crate is version 0.16.9, MIT-licensed. The engine corresponds to TeX Live 2021/2022-era support files in its bundles. Tectonic uses a release-automation tool ("Cranko") and publishes continuous builds plus a `curl|sh` installer.

**Core feature set:**
- **Self-contained / zero-config.** Linking the `tectonic` crate lets you "fully process TeX documents, from source to PDF, without relying on any externally installed software, configuration, or resource files," because it bundles the TeX tools and routes their I/O through a pluggable backend system.
- **On-demand bundle system + caching.** "Tectonic automatically downloads support files so you don't have to install a full LaTeX system… If you start using a new LaTeX package, Tectonic just pulls down the files it needs and continues processing." Files come from a "bundle" — a `.ttb` web bundle or legacy indexed tar/zip — hosted on infrastructure such as the Harvard Dataverse / Azure storage. Downloaded files are cached per-user (`tectonic -X show user-cache-dir`), and you can introspect bundles with `tectonic -X bundle search` and `tectonic -X bundle cat`.
- **Reproducible builds.** The bundle technology "allows for completely reproducible document compiles," and byte-for-byte reproducibility is an explicit project goal.
- **XeTeX engine features.** Modern fonts (TrueType/OpenType), Unicode input, and direct PDF output via xdvipdfmx.
- **Output formats.** PDF (default), XDV ("XeTeX DVI"), and SPX ("Semantically Paginated XDV"). Output format is selectable via `--outfmt`.
- **V2 CLI ("`tectonic -X`").** A "cargo-like" interface with subcommands anchored on a `Tectonic.toml` document definition: `new`, `init`, `build`, `compile` (freestanding, V1-like), `watch`, `show`, `bundle`, and `dump`. V2 is activated by a leading `-X` (or by running a binary whose name contains "nextonic"); V1 remains default and can be forced with `-Y`. `tectonic -X dump` "runs a partial document build and outputs a requested intermediate file," to help integrate external tools.
- **`Tectonic.toml` workflow.** Defines `[doc]` (name, bundle URL, extra search paths) and one or more `[[output]]` profiles (type = "pdf", `tex_format` defaulting to "latex", `shell_escape`, `shell_escape_cwd`, `synctex`, input files). `tectonic -X new` scaffolds a `src/` + `Tectonic.toml` project.
- **`watch` command.** Rebuilds on source change (built on the `watchexec` framework) and can run an executable on each rebuild; works best in the V2 `Tectonic.toml` model.
- **Bibliography.** A built-in BibTeX engine (as of 0.15.0 ported to pure Rust); biber is supported as an external tool, and the V2 CLI prefers a `tectonic-biber` binary if present. Tectonic parses the `.run.xml` to find which resources biber needs and auto-reruns TeX as needed.
- **Shell-escape.** Off by default for security; enabled per-output via `shell_escape = true` in `Tectonic.toml` or `-Z shell-escape`. An `--untrusted` flag and `TECTONIC_UNTRUSTED_MODE` env var harden against malicious inputs.
- **SyncTeX, makefile-rule emission, keep-intermediates/keep-logs** are all supported options.
- **HTML output (experimental).** Tectonic added an internal flag to emit SPX instead of XDV; the `spx2html` engine converts SPX to HTML, producing an asset-specification (JSON) that can be merged across compilations. The engine's "semantic pagination" mode is, per its own docs, "essentially unimplemented" / experimental. Related experimental projects: `tt-weave` (a WEAVE-like literate-programming tool that emits specialized TeX compiled to an HTML web app — used to produce the book *XeTeX: A Pseudoprogram*) and `Tectonopedia`.

**Architecture: the Rust crate graph.** The monolithic engine has been split into ~two dozen specialized crates. Foundational crates: `tectonic_errors` (error types), `tectonic_status_base` (status reporting), `tectonic_io_base` (the pluggable I/O abstraction implementing TeX's I/O model via an `IoProvider` trait), and `tectonic_bridge_core` (a framework to launch the unsafe C/C++ "engines" through FFI, with C headers generated by cbindgen). Major functional crates:
- `tectonic_engine_xetex` — "The XeTeX program as a reusable crate"; the core TeX engine.
- `tectonic_engine_xdvipdfmx` — the xdvipdfmx (XDV→PDF) engine.
- `tectonic_engine_bibtex` — the (now Rust) BibTeX engine.
- `tectonic_engine_spx2html` — SPX→HTML conversion.
- `tectonic_xdv` — a pure decoder for the XDV and SPX formats.
- `tectonic_docmodel` — the `Tectonic.toml` document model; notably **"only provides data access: you can use this crate without needing to link with the full Tectonic engines."**
- `tectonic_bundles` — the bundle (support-file) system.
- `tectonic_geturl`, `tectonic_xetex_layout`, `tectonic_xetex_format` (format-file introspection), `tectonic_pdf_io`, plus bridge crates for flate, freetype2, graphite2, harfbuzz, icu, png, fontconfig.

**Using Tectonic as a Rust library.** This is a first-class use case. The one-liner is `tectonic::latex_to_pdf(latex) -> Result<Vec<u8>>`. For realistic control, the `driver` module exposes a high-level processing session; lower-level structs (`TexEngine`, `XdvipdfmxEngine`, `BibtexEngine`, `Spx2HtmlEngine`) let you invoke individual stages. The I/O backend is pluggable, so you can feed source in memory and capture outputs without touching disk. Note the engines are not reentrant (you can't run the TeX and BibTeX engines simultaneously), and the C core is invoked via FFI.

### Part 2 — Does Tectonic fit the macro-expansion / parsing need? (Core analysis)

**Short answer: no.** Tectonic is an *execution* engine, not a *parsing/normalization* library. It can tell you whether a document compiles and give you the final typeset geometry, but it cannot hand you an expanded-yet-structured LaTeX token stream.

**No pre-layout, structure-preserving IR exists.** Walking Tectonic's pipeline: source → XeTeX interpreter (mouth/gullet/stomach: tokenization, macro expansion, primitive execution, box building) → XDV/SPX → xdvipdfmx → PDF. The only intermediate artifacts you can extract (via `--outfmt` or `tectonic -X dump`) are XDV or SPX. Both are DVI derivatives: per the `tectonic_xdv` crate, "both of these file formats are derived from the venerable 'device independent' (DVI) format used by TeX. The XDV format… adds a few codes needed to express native fonts." SPX ("semantically-paginated XDV") "is essentially the same as XDV, but expresses output that is not paginated for print — this is what Tectonic uses to produce its HTML output."

**Why XDV/SPX is too low-level for LaTeX→Typst.** XDV/SPX operate at the level of glyph positioning, font selection, boxes, rules, and `\special` commands — coordinates and character codes on a page. By the time the engine emits them, `\section{Intro}` has become "set these glyphs in this font at these positions, plus a vertical skip and a counter side-effect"; an `align` environment has become positioned math glyphs; `\emph{x}` has become a font switch. The logical tree (sections, lists, theorem environments, the *semantics* of a fraction or an integral) is irretrievably gone. Even Tectonic's own HTML effort underscores this: to get good HTML the team had to invent SPX *and* note that "to achieve the best HTML output, the engine will have to emit intermediate data that are incompatible with XDV" — and even then they re-inject structure through `\special{tdux:...}` markers manually placed in macros (as seen in `tt-weave`'s preamble). In other words, the structure has to be *put back in by hand*; it is not recoverable from the geometry. Driving a Typst conversion that must preserve `\section`, environments, and math semantics from XDV/SPX would mean reverse-engineering document logic from positioned glyphs — strictly harder than parsing the source.

**The fundamental TeX problem (applies to any engine, including Tectonic).** Even setting Tectonic aside, "run TeX and dump fully-expanded LaTeX" is not generally well-defined:
- **Turing-completeness.** TeX's macro language is Turing-complete; expansion can loop, compute, and branch (`\expandafter`, `\csname`, `\if…\fi`, `\futurelet`, etc.). There is no guarantee a "fully expanded" normal form exists or terminates.
- **Expansion is intertwined with execution and mutable catcodes.** TeX broadly separates an *expansion* stage (the "gullet") from *execution* (the "stomach"), but they interact: macros can change category codes mid-stream, so the very tokenization of later input depends on side effects of executing earlier tokens. The classic `\catcode\`\$=11` example shows a character's catcode is fixed at tokenization time and cannot be cleanly "expanded away" afterward.
- **The e-TeX "tricks" are partial at best.** `\detokenize` yields catcode-12 character tokens for a *balanced text* but does not deeply expand; `\scantokens` re-reads characters under the current catcode regime (using `\input`'s mechanism); `\edef`/`\message`/`-output-comment` and `\tracingmacros` can surface *some* expansions but stop at unexpandable primitives and do not produce reusable, structured LaTeX. None of these gives you "all packages and macros unfolded into clean LaTeX" reliably.
- **The practical corollary** (seen in real tools): you do *not* want to fully expand structural/semantic commands. Pandoc, for instance, deliberately lets you disable macro expansion (`latex-latex_macros`) because "some macros only modify things in included packages, which pandoc's macro expansion can't modify," and fully expanding `\section` into kernel primitives would destroy exactly the structure a converter needs. The right move is *selective* expansion of user-defined shorthand macros while *preserving* structural/semantic commands as recognizable nodes — which is a source-level parsing task, not an engine-execution task.

### Part 3 — Comparison of alternative tools

**`mitex-parser` / `mitex` / `tylax` (Rust) — best fit.** MiTeX is "LaTeX support for Typst, powered by Rust and WASM" (Apache-2.0; current package version 0.2.7 on Typst Universe, with the `mitex-parser` crate at 0.2.4 on docs.rs). Critically for you, its parser is a standalone, published crate separate from the converter. Its documentation states it "returns an untyped syntax node representing the AST defined by `rowan`" — i.e. a **lossless concrete syntax tree built on the same `rowan` library rust-analyzer uses**, convertible to typed nodes or to a mutable cursor tree. The primary entry point is `parse(text, spec)`, where `spec` is a `CommandSpec` describing each command's argument shape; without a spec, "all commands are parsed as bare names without arguments." It supports user-defined macros (`\newcommand{\mysym}{\alpha}`, delimited parameters `#1…#2`) and partial document/text-mode parsing (`\section`, `\textbf`, `\emph`, `itemize`/`enumerate`, `figure`/`table`/`tabular`, inline/block math), though text-mode coverage is explicitly "in development" and weaker than its math support. MiTeX is small and fast — per its README, "MiTeX has a size of just about 185 KB, comparing that texmath has a size of 17 MB," and "To convert 32.5k equations from OI Wiki, texmath takes about 109s, while MiTeX WASM takes only 2.28s and MiTeX x86 takes merely 0.085s."

**`tylax` (Rust) — a near-exact precedent for your project.** Tylax is "a bi-directional converter between Typst and LaTeX" (Apache-2.0, crate `tylax` ~0.3.x), available as a CLI (`t2l`) and library (`latex_to_typst`, `latex_document_to_typst`, etc.). Its architecture is **Macro Engine → MiTeX Parser → AST → Converter**: it *uses* `mitex-parser` for LaTeX→AST and adds its **own token-based macro expander** that runs first. Per its changelog, v0.2.0 "replaced the old regex hacks with a proper token-based macro expander," supporting `\def`, `\newcommand`, `\let`, `\newif`, delimited parameters, `\begingroup` scoping, recursion limits, and dynamic `$` tracking for `\ifmmode`. Its author started Tylax precisely "because… existing regex-based scripts [were] too fragile for nested environments" — the same motivation Zeyu has. Caveats from its own docs: the crate docs more modestly call it "Basic LaTeX macro definition and expansion"; it had to patch infinite-recursion on mutually-recursive macros and macro-definition leakage; TikZ↔CeTZ is experimental; and it is "static" (no full dynamic evaluation). **This is both your strongest reusable dependency and your strongest reference implementation — you can read exactly how it expands macros over the mitex AST.**

**`tex2typst-rs` (Rust).** A Rust library converting TeX→Typst aimed at *idiomatic, readable* Typst (vs MiTeX's render-fidelity goal). It exposes `tex2typst`, `text_and_tex2typst`, and `tex2typst_with_macros(tex, macros)` (accepting `\newcommand`/`\def` definitions), plus a parse→convert→write pipeline (`parse_tex` → `convert_tree` → `TypstWriter`). License is GPL-3.0-or-later — note the copyleft implication for a CLI you may distribute. Math-focused; less document-structure coverage than tylax.

**Pandoc LaTeX reader (Haskell).** Pandoc has a mature LaTeX reader that parses to a structured AST and a Typst *writer* (so `pandoc -f latex -t typst` exists today). It performs *selective* macro handling: with the `latex_macros` extension it parses `\newcommand` definitions and applies them to math/raw LaTeX, but it does **not** implement full TeX. Known limitations are instructive: macros with parameters must be declared with the right arity or you get parse errors; macro expansion can mishandle optional arguments, eat newlines, add stray `{}` braces in math, and fails to expand macros inside some environments like `align`. Integration cost from Rust = shelling out to the `pandoc` binary (no native crate). Good as a baseline/fallback converter, but you'd be at the mercy of its LaTeX coverage and AST, and Pandoc itself warns its macro expansion "can't modify" package internals.

**LaTeXML (Perl).** LaTeXML is "a Perl reimplementation of TeX's parsing and digestion algorithm coupled with a customizable XML emitter." It models catcodes, tokens, a Mouth, Gullet (macro expansion), Stomach (primitives), and Constructors that emit XML — explicitly "geared towards preserving macro semantics." It produces a semantic, LaTeX-near XML tree (sections, math as presentation/content MathML), and is what arXiv uses for its HTML papers. arXiv's own reporting quotes LaTeXML/ar5iv developer Deyan Ginev: "In 2022, we can report 74% of sources convert successfully, with 97% of articles at least partially viewable" (1.78 million pages) — and arXiv frames the ~97% figure as "an availability ceiling rather than a quality claim," with roughly "25% of papers [having] conversion errors" and "only 3% totally fail conversion." This is the gold standard for *structure-preserving* conversion, and its XML would be an excellent IR to map to Typst. Costs: (1) it's Perl, so integration = shelling out to the `latexml`/`latexmlpost` binaries (heavy dependency; conversion ranges from ~30 ms per formula to minutes per book); (2) it "does not use the actual TeX compiler… emulating just enough of TeX to preserve the information provided by the author," so packages with semantic intent need hand-written LaTeXML "bindings" — the team has implemented special cases "for more than 400 of the most commonly used LaTeX packages in arXiv, but that still leaves a considerable number of packages it does not recognize"; (3) it is a large, complex system to emulate or depend on. Use it as a reference for *how* to preserve semantics, and possibly as an optional high-fidelity backend, but not as an embeddable Rust library.

**`unified-latex` (JavaScript/TypeScript).** A PEG.js-based LaTeX parser producing an AST, with a rich utility ecosystem (`unified-latex-util-parse`, `-util-macros`, `-util-split`, `-to-hast`). It explicitly acknowledges that "parsing LaTeX isn't possible since it effectively has no grammar" and "makes practical assumptions," failing on things like redefined control sequences or complex TeX macros. It supports `\makeatletter`/`expl3` flag modes and macro-info records describing argument shapes — conceptually very similar to mitex's command spec. Costs: it's JS/TS, so for a Rust CLI you'd either shell out to Node or port concepts. Good *design reference* (especially its macro-info/argument-shape model and AST-manipulation utilities), but not a native Rust dependency.

**`latexpand` and `de-macro` (Perl/Python) — flattening utilities, not parsers.** `latexpand` "flatten[s] LaTeX file by expanding `\include` and `\input`, … and remove[s] comments," with `--expand-usepackage` to inline `.sty` files, `--expand-bbl`/`--biber` to inline bibliography, and `-d key=val` to define simple macro replacements. It does *textual* substitution (and even has bugs around verbatim environments) — it is not a parser and does not understand macro semantics. `de-macro` "expand[s] macros defined in `\(re)newcommand` or `\(re)newenvironment`… in the document, or in the document's 'private' package file" — i.e. it expands *your private* macros while leaving package/structural commands alone, which is exactly the selective-expansion behavior you want for the `\input`/`\usepackage` flattening and private-macro stage. (Related: `xpandlatex` does macros+refs+cites+includes with `%XP` directives; `flattenPaper` orchestrates latexpand+de-macro for arXiv submission.) Integration = shelling out; they're small and well understood. Use them for the *flattening* stage, not the structural parse.

**Generic Rust parser toolkits (`nom`, `chumsky`, `pest`, `lalrpop`).** If you build your own grammar, `chumsky` is the best fit for a user-facing language parser (expressive combinators, first-class error recovery, can parse token trees) and is what you'd reach for to write a hand-rolled LaTeX-ish tokenizer/parser; `nom` is leaner and byte-oriented. But writing a LaTeX parser from scratch duplicates exactly what `mitex-parser` already did with `rowan` — only worthwhile if mitex's command-spec model proves too limiting. There is an old `latex` crate but it is minimal and not a serious document parser.

## Recommendations

**Verdict on Tectonic for the "expand packages and macros" step: do not use it for that.** Tectonic runs the full TeX engine and only emits geometry-level XDV/SPX or final PDF; it has no API surface that yields an expanded, structure-preserving LaTeX token stream. Using it as your normalizer would require reconstructing document logic from positioned glyphs — strictly harder than parsing the source, and it inherits TeX's Turing-complete, catcode-entangled expansion model that makes "expand everything to clean LaTeX" ill-defined anyway.

**Recommended architecture for the Rust CLI converter (staged):**

1. **Stage 0 — Optional validate/resolve (Tectonic here).** Before converting, optionally call `tectonic::latex_to_pdf` (library API) on the input. If it fails to compile, surface the TeX error and stop — you don't want to convert a broken document. This also primes Tectonic's bundle cache so package `.sty` sources are available locally for later inspection (`tectonic -X bundle cat`).

2. **Stage 1 — Flatten.** Run `latexpand` (and/or `de-macro`) to inline `\input`/`\include`, optionally expand `\usepackage` for local style files, inline the `.bbl`, and strip comments — producing a single self-contained `.tex`. This is textual and safe; shell out to the binary. (You may later reimplement the `\input`-flattening natively in Rust to drop the Perl dependency.)

3. **Stage 2 — Parse to a lossless AST.** Feed the flattened source to **`mitex-parser`**, getting a `rowan` syntax tree. Supply/extend a `CommandSpec` so your structural and package commands parse with correct argument shapes. This is your normalized, structured IR — vastly better than regex.

4. **Stage 3 — Selective macro expansion.** Reuse or mirror **Tylax's macro engine**: expand *user-defined* macros (`\newcommand`, `\def`, `\let`, `\newif`, delimited params, `\begingroup` scoping, `\ifmmode` math-mode tracking) over the token/AST level, with a hard recursion/step limit — while **preserving** structural/semantic commands (`\section`, environments, `\frac`, etc.) as first-class AST nodes to be mapped to Typst. Do not expand the structural/semantic commands.

5. **Stage 4 — Map AST → Typst.** Walk the expanded AST and emit Typst, leaning on MiTeX's command-spec/conversion mappings for math and Tylax's document-level mappings (sections, lists, tables, references). For inline/block math specifically, MiTeX is the proven engine.

6. **Stage 5 — AI fallback (SKILL.md).** When the parser hits an unknown command/environment, a package-specific construct, or a low-confidence mapping, route *that fragment* (with surrounding context) to the AI agent guided by your SKILL.md files. Keep the AI scoped to corner cases — not the whole document — so results stay deterministic and reviewable.

7. **Stage 6 — Round-trip validation (Tectonic again).** Optionally compile the *original* LaTeX with Tectonic and the *generated* Typst with the Typst compiler, render both to PDF/PNG, and diff them (visually or via image similarity) as an automated regression/acceptance oracle. This is where Tectonic genuinely adds value: as a trusted reference renderer embedded directly in your Rust binary.

**Benchmarks/thresholds that would change the plan:**
- If `mitex-parser`'s text-mode coverage proves too thin for your corpus, escalate to **LaTeXML** as an optional high-fidelity backend (shell out; map its semantic XML to Typst) — accept the Perl dependency and slower speed for hard documents.
- If you need broad format coverage fast and can tolerate Pandoc's macro limitations, wire **`pandoc -f latex -t typst`** as a fallback converter and compare its output against your native path.
- If GPL is acceptable, evaluate **`tex2typst-rs`** for more idiomatic/readable Typst math output.
- If your documents rely heavily on bespoke package semantics, invest more in the SKILL.md/AI layer and in extending the `CommandSpec`, rather than trying to handle them in the parser.

## Caveats
- **TeX has no real grammar.** Every source-level parser (mitex, unified-latex, Pandoc) makes "practical assumptions" and will fail on adversarial input: redefined control sequences, catcode changes, `\makeatletter`/`expl3` regions, and deeply programmatic macros. A hybrid parser-plus-AI design is the pragmatic response precisely because perfect parsing is impossible.
- **Selective vs. full expansion is the crux.** Fully expanding macros (even if it terminated) would destroy the structural/semantic commands you need to preserve. Expand only user shorthand; keep structure.
- **Math-mode nuances.** Spacing, `\left/\right` vs `\middle`, optional-argument handling, and brace-grouping differences are where converters most often go wrong (Pandoc's stray-`{}` and `align`-expansion bugs are cautionary). MiTeX's fidelity focus exists for this reason.
- **Maturity and licensing of dependencies.** `mitex-parser`/`tylax` are pre-1.0 (0.2.x/0.3.x), Apache-2.0, with acknowledged uncovered edge cases; `tex2typst-rs` is GPL-3.0; LaTeXML/Pandoc/latexpand are external-process dependencies (Perl/Haskell). Pin versions and budget for upstream churn.
- **Tectonic's role is bounded.** It is an excellent embeddable compiler and validation oracle, and its bundle system is a convenient package-source resolver — but it contributes nothing to the parsing/expansion step itself. Don't let its impressive engine tempt you into trying to extract structure from XDV/SPX.
- **Source-quality note.** Several capability claims (MiTeX's speed, Tylax's "full expansion support") come from project READMEs/marketing; the crates' own API docs are more conservative ("basic … expansion"). Treat headline numbers as indicative, and validate against your own corpus.
